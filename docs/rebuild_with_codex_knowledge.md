# SplitLog クロスプラットフォーム再構築ナレッジ

作成日: 2026-07-01

## 目的

この文書は、既存macOS版SplitLogをFlutterで再構築し、macOS、Windows、iPhone、Androidへ展開していく過程で得たナレッジを残すためのもの。

対象は、実装そのものの詳細ではなく、以下のような再利用しやすい知見とする。

- どの順番で意思決定すると進めやすかったか
- Codexへどう指示すると期待に近い結果になったか
- どの確認が問題発見に効いたか
- Flutterでデスクトップ/モバイルアプリを作るときに注意すべきこと
- Web開発経験者がネイティブアプリ開発へ入るときに詰まりやすい点

## 全体の進め方で効いた判断

### 既存アプリを直接改修せず、新規Flutterプロジェクトに分けた

今回もっとも大きな判断は、既存のSwift/AppKit版SplitLogを直接Flutter化するのではなく、`splitlog_x` という別プロジェクトを作ったこと。

これは正しい判断だった。

理由:

- 既存アプリを壊すリスクを避けられた
- Swift/AppKit版を参照実装として残せた
- Flutter版の構成を最初からクロスプラットフォーム前提にできた
- 大きな作り替えでもgit履歴やファイル構成が混ざりにくかった
- UI/UX比較時に旧アプリを実行して正とできた

学び:

既存アプリの技術スタックを大きく変える場合、既存コードを直接置換するより、新規プロジェクトで再構築し、旧アプリを「仕様書兼参照実装」として使う方が安全。

### 最初に方針を確定してから実装した

実装に入る前に、以下を先に決めた。

- データ同期はv1では行わず、端末ローカル保存のみ
- 旧データ移行は手動インポート必須 + macOSのみ自動検知して確認表示
- 保存方式はJSONファイル
- macOSはメニューバー常駐
- Windowsはタスクトレイ常駐
- iPhone/Androidは全画面アプリ
- モバイルは縦画面固定
- モバイルv1では常時バックグラウンド実行や通知欄常時表示はしない
- macOS配布はまずzip/dmg、Windowsはzip/msi、iOSはTestFlight、AndroidはAPK

これにより、途中で「どこまで作るか」「何を後回しにしてよいか」の判断がしやすくなった。

学び:

クロスプラットフォーム化では、実装前に「同じにするもの」と「プラットフォーム別に変えるもの」を明確に分ける必要がある。

## Codexへの指示で特に効果があったもの

### 「コード修正は禁止」と明示する

調査段階では、最初に「コードの修正は禁止」と伝えたことが有効だった。

効果:

- いきなり実装へ進まず、現在の構造確認に集中できた
- 既存アプリの仕様、ブランチ状態、変更差分を安全に確認できた
- 方針検討と実装作業を分離できた

再利用できる指示:

```text
まず現在の実装を確認してください。
この段階ではコード修正は禁止です。
構成、依存、保存方式、UI、リスクだけを整理してください。
```

### 「憶測で修正せず、旧実装と現在実装を確認して」と繰り返す

UI再現で特に重要だった指示。

今回、円リング、フォント、メニュー、Split名編集、ポップアップ、セッション切り替えなどで、最初はそれっぽい実装になっても、旧SplitLogとは細部が違うことがあった。

そこで、以下の指示が効いた。

```text
憶測で修正せず、必ずSplitLogの実装を確認し、
必要があればスクリーンショットを撮り、
現在の実装と照らし合わせて修正してください。
```

効果:

- Codexが一般的なUI知識だけで補完するのを防げた
- Swift/AppKit版の実装を正として比較できた
- スクリーンショット、旧コード、Flutter実装の三点比較が定着した
- 細かな見た目のズレを潰しやすくなった

学び:

既存アプリの再現では「よくあるUI」ではなく「このアプリのUI」が正になる。AIには、一般解ではなく参照実装を優先するよう明示する必要がある。

### 1回で大量に直させず、気になる点を番号付きで渡す

ユーザー側から、問題点を以下のように番号付きで渡したことが有効だった。

例:

- Splitクリック時の青いフォーカス枠
- Split名クリック編集
- セッション名クリック編集
- セッション一覧からの切り替え
- ヘルプ項目クリック時の説明表示
- 全体時間とSplit合計の不一致
- メニュー画面の設定変更不可

効果:

- 修正漏れを確認しやすい
- 後回しにするものも理由付きで整理できる
- 作業単位が明確になり、完了確認がしやすい

再利用できる指示:

```text
気になる点を番号で列挙します。
それぞれについて、今回修正するか、後回しにするなら理由を出してください。
修正時は既存実装を確認してから進めてください。
```

### 「動作やUIを変えずに掃除して」と伝える

Stepの区切りで不要コード確認を依頼したことも有効だった。

効果:

- 試行錯誤の残りを整理できた
- 不要コード削除と機能変更を混ぜずに済んだ
- Step完了前の品質確認になった

再利用できる指示:

```text
動作やUI、処理は絶対に変えずに、不要なコードや残骸だけ確認して削除してください。
不要なものがなければ、ないと言ってください。
```

## UI再現で得た知見

### サンプル画像を正にしない判断が重要だった

旧リポジトリに `SplitLog/sample/*.png` があったが、これは完成版UIではなく、旧SplitLog制作時の参考画像だった。

そのため、再現基準から外した。

最終的な基準:

- 見た目: 旧macOS版を実行して取得したスクリーンショット
- 挙動: Swift/AppKitコードとREADME
- 実行できない場合: Swift/AppKitコードから構造を読み取る

学び:

リポジトリ内に画像があっても、それが現在仕様とは限らない。必ず「この画像は正か」を確認する。

### 細部の違いは実際に触らないと見つからない

静的UIが似ていても、実際に触ると以下のような差分が出た。

- クリック時に不要な青いフォーカス枠が出る
- Split名編集で長い名前が欠ける
- 折り返しテキストの編集初回表示がおかしい
- セッション一覧ポップアップが外クリックで閉じない
- ロックON時に他ウィンドウより前面に出ない
- モノクロモードでも一部ボタンが青い
- サマリーの切替ボタンが固定表示のまま

学び:

UI再現はスクリーンショット一致だけでは足りない。クリック、編集、外クリック、長文、モード切替、ポップアップ、終了操作まで触って確認する必要がある。

### Flutter UIだけでは完結しない部分がある

macOS版では以下のような処理にSwift/AppKit実装が必要だった。

- メニューバー常駐
- ステータスアイコン
- 外クリックで非表示
- ウィンドウの前面固定
- 閉じるボタンの扱い
- グローバルショートカット
- ファイル選択ダイアログ
- メール作成画面の起動

学び:

FlutterはUIと共通ロジックをかなり共有できるが、OSの常駐、ウィンドウ制御、ショートカット、配布まわりはネイティブコードが必要になることがある。

これはFlutterとして失敗ではなく、クロスプラットフォームアプリでは自然な構成。

## ロジックと保存方式の知見

### v1はJSON保存で十分だった

SplitLogのデータ量と用途を考えると、v1でSQLiteを導入する必要はなかった。

JSON保存を選んだ理由:

- 旧SplitLogの `sessions.json` と互換を持たせやすい
- データ構造が比較的単純
- ローカル保存のみなら十分
- デバッグしやすい
- インポート/エクスポートしやすい

学び:

アプリだからといって最初からDBを入れる必要はない。データ量、検索要件、同期要件が小さい場合は、JSONファイルの方が安全で速いことがある。

### 旧データ移行は「自動 + 手動」の両方が必要

macOS版では旧SplitLogの標準保存先を検知して確認ダイアログを出す方針にした。

ただし、自動検知だけにはしなかった。

理由:

- 保存先が変わっている可能性がある
- サンドボックスや権限で読めない可能性がある
- 他ユーザー環境では検知できない場合がある
- WindowsではmacOS旧保存先の自動検知は意味が薄い

そのため、手動で `sessions.json` を選ぶ導線を残した。

学び:

データ移行は自動化だけに寄せると失敗時の逃げ道がなくなる。手動インポートを必ず用意する。

### パスの確認は配布前に必要

旧データ検知で `/Users/hamada` に固定されていないか確認した。

結果:

- 実装は `HOME` から組み立てていた
- macOSのentitlementsもhome-relative-pathだった
- ユーザー名には依存していなかった
- 依存していたのは旧アプリの保存仕様、つまり `SplitLog` と `hamachi.SplitLog`

学び:

配布前には、自分のユーザー名、絶対パス、開発環境固有のパスが混入していないか確認する。

## 環境構築の知見

### `flutter doctor` を完了条件にしたのが良かった

最初は `flutter` コマンドが存在しない状態だった。

その後、以下を整備した。

- Flutter SDK
- Android Studio
- Android SDK
- Android SDK Command-line Tools
- Android licenses
- Xcode追加コンポーネント
- CocoaPods

最終的に `flutter doctor` が `No issues found!` になるまで確認した。

学び:

Flutter開発では、実装前に `flutter doctor` を通すのが実質的な環境構築の完了条件になる。

### Android StudioはIDEとして必須ではないが、SDK管理には必要

VSCodeを使う場合でも、Android SDKやエミュレータ、cmdline-toolsを入れるためにAndroid Studioは必要になった。

学び:

FlutterのIDEはVSCodeでよい。ただしAndroid向けビルドにはAndroid StudioがSDK管理ツールとして必要。

### Flutterプロジェクト名とアプリ表示名は別

`flutter create SplitLog_x` はDartパッケージ名として不正だった。

Flutterプロジェクト名は `splitlog_x` のように小文字 + アンダースコアにする必要がある。

一方で、実際のアプリ表示名は後から `SplitLog` に変更できる。

学び:

Flutterのプロジェクト名、Dartパッケージ名、アプリ表示名、Bundle IDは別物として考える。

## 配布の知見

### App Storeに出さなくても、正式配布にはDeveloper Programが必要

macOS版をzipで身内配布することはできた。

ただし、Apple Developer Programに入っていない場合、Developer ID署名とnotarizationはできない。

そのため、現時点の配布は以下の位置づけになる。

- 身内確認用zip: 可能
- Gatekeeper警告なしの一般Web配布: Developer ID署名 + notarizationが必要
- Mac App Store配布: Apple Developer Program + App Store向け設定が必要

学び:

App Storeに出さない場合でも、macOSで一般ユーザーに自然に配布するならDeveloper IDと公証が必要。

### zip作成はFinderではなく `ditto` が安全

macOSアプリは `.app` がディレクトリ構造を持つため、配布zipは `ditto` で作る方が安全。

使用した形:

```bash
flutter build macos --release
mkdir -p dist
ditto -c -k --sequesterRsrc --keepParent \
  build/macos/Build/Products/Release/SplitLog.app \
  dist/SplitLog-macOS-v1.0.0.zip
```

学び:

macOSアプリ配布では `.app` の構造やリソースを壊さない作り方を選ぶ。

## Step分割で効いたこと

今回のStep1は以下の順で進めた。

1. 旧SplitLogのUI/機能棚卸し
2. Desktop静的UI再現
3. コアモデル/タイマーロジック
4. UIとロジック接続
5. macOSメニューバー常駐/小型ウィンドウ化
6. JSON保存と旧データインポート
7. macOS版完成度チェックと残タスク修正
8. 表示名、アイコン、配布zip確認

この順番は有効だった。

理由:

- 先に見た目を合わせることで、ユーザーが早い段階で違和感を指摘できた
- ロジックを後から接続したことで、UI差分と機能差分を分けて見られた
- 常駐化や配布はOS依存が強いため、後半に回した方がよかった
- Stepごとにコミットでき、戻しやすかった

学び:

既存アプリ再現では、以下の順が進めやすい。

```text
調査
-> 静的UI
-> コアロジック
-> UI接続
-> OS固有機能
-> 保存/移行
-> 完成度チェック
-> 配布
```

## AIに依頼するときの実用テンプレート

### 調査だけしてほしいとき

```text
まず現在の実装を確認してください。
この段階ではコード修正は禁止です。
構成、保存方式、UI、OS依存処理、リスクを整理してください。
```

### 方針を固めたいとき

```text
実装前に方針を決めます。
未確定な点、判断が必要な点、後から変えると重い点があれば質問してください。
認識違いをなくしてから実装に進みたいです。
```

### 既存アプリの再現を依頼するとき

```text
憶測で実装せず、必ず旧アプリの実装、スクリーンショット、現在のFlutter実装を比較してください。
見た目は旧アプリ、挙動は旧コードを正とします。
違いがある場合は、どちらを優先したか理由も出してください。
```

### 問題点をまとめて直してほしいとき

```text
気になる点を番号付きで列挙します。
それぞれについて、今回修正するか、後回しにするなら理由を提示してください。
修正時は既存実装を確認してから進めてください。
```

### Step完了前に掃除したいとき

```text
今回の修正で不要なコードや試行錯誤の残りがないか確認してください。
動作、UI、処理は絶対に変えず、不要な部分だけ削除してください。
なければないと言ってください。
```

### 配布前に確認したいとき

```text
配布前に、アプリ名、アイコン、Bundle ID、署名状態、保存先、ユーザー固有パス混入、zip作成手順を確認してください。
正式配布と身内配布で違う点も整理してください。
```

## 今後のStep2以降に活かすこと

### Windows対応で注意すること

macOSのメニューバー常駐に相当するものは、Windowsではタスクトレイ常駐。

方針:

- タスクバー常駐ではなく、通知領域/タスクトレイを使う
- Desktop UIはmacOS版を共有する
- ウィンドウ制御、トレイ制御、終了処理はWindows固有実装として分離する
- まず通常ウィンドウで動作確認し、その後タスクトレイ化する

注意:

- Windows実機またはWindows環境での確認が必要
- macOS上ではWindowsのトレイ挙動を完全には確認できない
- Flutterプラグイン導入時は公式ドキュメントと保守状況を確認する

### Mobile対応で注意すること

iPhone/Androidでは、macOS版の横長ポップアップUIをそのまま使わない。

方針:

- 縦画面固定
- 全画面アプリ
- Desktop UIの情報設計を保ちつつ再配置
- 常時バックグラウンド実行はv1では狙わない
- タイマーは開始時刻/累積時間から復帰時に正確に復元する
- 通知欄/ロック画面の常時表示は後回し

注意:

- モバイルでは「見た目の再現」より「同じ概念をスマホで自然に使えること」が重要
- 片手操作、タップ領域、スクロール、モーダル設計を優先する

## 今回の一番大きな学び

AIに任せるときは、単に「作って」ではなく、以下を明確にすると成果が安定する。

- 何を正とするか
- 何を変えてよいか
- 何を変えてはいけないか
- どの順番で進めるか
- いつ確認するか
- どこまでをStep完了とするか
- 不明点を先に質問させるか

今回うまくいった理由は、実装そのものだけではなく、ユーザー側が途中で違和感を細かく指摘し、Codex側が旧実装と照合しながら修正する流れを作れたことにある。

特に重要だった姿勢:

```text
既存アプリ再現では、AIの推測より既存実装を優先する。
細部の違和感は放置せず、その場で潰す。
方針が未確定なまま実装しない。
Stepごとに完了条件を確認する。
```

この進め方は、今後のWindows版、モバイル版、配布整備でもそのまま使う。
